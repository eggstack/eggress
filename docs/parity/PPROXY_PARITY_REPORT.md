## Protocol-Crate-Only Caveats

The following capabilities are implemented in protocol crates but
**refused by the runtime/config compiler** (Phase 25-28 H5/H6/H7).
They cannot be promoted to `drop_in` until the config compiler and
runtime supervisor accept them:

| ID | Protocol crate | Refused by |
|----|---------------|------------|
| `uri.scheme_raw` | eggress-protocol-raw | config + runtime |
| `uri.scheme_tunnel` | eggress-protocol-raw | config + runtime |
| `uri.scheme_ws` | eggress-protocol-websocket | config + runtime |
| `uri.scheme_wss` | eggress-protocol-websocket | config + runtime |
| `uri.scheme_h2` | eggress-protocol-http | config + runtime |
| `protocol.ws_runtime` | eggress-protocol-websocket | config + runtime |
| `protocol.raw_runtime` | eggress-protocol-raw | config + runtime |
| `protocol.h2_runtime` | eggress-protocol-http | config + runtime |

These are correctly classified as `native_equivalent` in the manifest.
Promotion to `drop_in` requires completing the config compiler integration
and runtime supervisor wiring.

## Python Drop-In Status

All 10 Python `drop_in` capabilities have integration test evidence.
The Python binding layer wraps `eggress-embed` via PyO3 with GIL release
on all blocking calls.

The `python.async_wrappers` capability is marked `unsupported` — the
`AsyncEggressHandle` exists but is not a full async wrapper API.

## Next Recommended Phase

1. **Wire protocol-crate-only features through config compiler/runtime**: Promote
   `uri.scheme_ws`, `uri.scheme_h2`, `uri.scheme_raw` and their runtime counterparts
   from `native_equivalent` to `drop_in`.

2. **Add Trojan server**: Promote `protocol.trojan_server` from `unsupported` to
   at least `native_equivalent`.

3. **SOCKS5 UDP framing alignment**: Investigate whether pproxy's custom framing
   can be matched or whether the `compatible_with_warning` classification is
   permanent.

4. **Differential tests for Shadowsocks TCP/UDP**: Current evidence is `integration`
   (interop with standard Shadowsocks); add pproxy-differential tests.

5. **Python async wrapper API**: Complete the `python.async_wrappers` capability.

6. **CI integration**: Wire `validate_pproxy_parity_manifest.py` into the CI
   pipeline as a required check.
