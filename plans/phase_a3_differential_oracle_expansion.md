# Phase A3 — Differential oracle expansion

## Objective

Turn the existing pproxy differential harness into a release-grade, data-driven compatibility oracle. The oracle must compare observable behavior rather than only successful startup or payload echo, and every release-blocking `drop_in` claim must reference an oracle scenario or a documented reason why differential testing is impossible.

## Dependencies

- A1 establishes accurate capability classifications.
- A2 establishes normalized composition IDs and matrix cells.

The oracle should consume those stable IDs rather than inventing another capability vocabulary.

## Reference versions

Pin `pproxy==2.7.9` as the primary baseline. Store the exact Python version, dependency lock information, Eggress commit, target triple, and scenario schema version in every report. A separate non-blocking job may run against the latest pproxy release.

## Workstream 1: scenario schema

Define a declarative scenario format, preferably TOML or JSON, with fields for:

- scenario ID and description;
- capability and composition IDs;
- supported platforms;
- pproxy and Eggress command lines;
- environment variables and fixtures;
- listener readiness strategy;
- client action sequence;
- expected semantic outcome;
- output normalization rules;
- wire/transcript capture settings;
- timeout budgets;
- cleanup expectations;
- comparison mode: exact, normalized semantic, wire-level, or interoperability-only;
- expected divergence IDs where compatibility is intentionally approximate.

The schema must be versioned and validated before any child process starts.

## Workstream 2: robust process supervisor

Build on the existing oracle process runner and require:

- deterministic free-port reservation or readiness coordination;
- process-group/job-object ownership so descendants are killed;
- bounded stdout and stderr capture;
- independent startup, scenario, idle, and shutdown timeouts;
- readiness probes appropriate to TCP, UDP, Unix, TLS, and protocol-specific listeners;
- structured exit information;
- cleanup on normal completion, timeout, panic, test cancellation, and guard drop;
- no dependence on pproxy emitting a startup banner;
- artifact retention for failed runs.

Use platform abstractions for Unix process groups and Windows job/process handling. Avoid shell invocation where direct argv execution is possible.

## Workstream 3: semantic observations

Capture and normalize:

- actual bound addresses;
- stdout and stderr records;
- exit code or terminating signal;
- connection success/failure category;
- protocol replies and status codes;
- bytes transferred in each direction;
- TCP half-close behavior;
- persistent-connection behavior;
- timeout and refusal timing within tolerances;
- authentication success/failure;
- route/upstream choice where observable;
- UDP source/destination and framing;
- shutdown and child-process cleanup;
- file/socket artifacts left behind.

Do not require byte-for-byte text equality for unstable human messages. Use stable semantic extraction with raw artifacts retained.

## Workstream 4: wire and transcript capture

Provide reusable client probes for:

- HTTP CONNECT;
- ordinary HTTP forward requests;
- SOCKS4 and SOCKS4a;
- SOCKS5 CONNECT and auth;
- SOCKS5 UDP ASSOCIATE;
- standalone UDP;
- TLS listeners;
- Shadowsocks and Trojan where feasible;
- reverse/backward control and data paths;
- Unix sockets;
- future WS/raw/H2/SSH/QUIC scenarios.

Prefer protocol-aware transcripts generated by test clients. Packet capture may be optional and environment-gated. Ensure secrets are redacted from retained artifacts.

## Workstream 5: initial scenario expansion

At minimum add scenarios for:

### CLI/process

- no-argument defaults;
- help/version;
- invalid flag and invalid URI;
- bind failure;
- verbosity/output destination;
- `--ssl`, block/rulefile, alive, PAC, test, sys, get, reuse, and daemon classifications;
- signal-based shutdown where applicable.

### HTTP

- CONNECT IPv4/IPv6/domain;
- auth success/failure;
- refusal and timeout;
- fragmented request;
- half-close;
- GET/POST/HEAD;
- content-length and chunked bodies;
- chunk extensions and trailers;
- persistent sessions and connection close;
- malformed headers and premature EOF;
- upstream close.

### SOCKS

- SOCKS4 IPv4/user ID/refusal/malformed;
- SOCKS4a domain;
- SOCKS5 IPv4/IPv6/domain;
- auth success/failure;
- malformed negotiation/request;
- refusal and half-close;
- UDP association lifecycle and framing.

### Chains/routing

- HTTP→HTTP;
- HTTP→SOCKS5;
- SOCKS5→HTTP;
- SOCKS5→SOCKS5;
- representative 3-hop chain;
- block and rulefile decisions;
- first-available, round-robin, random with statistical tolerance, and health/fallback behavior.

### Other current runtime

- Shadowsocks TCP/UDP external interoperability where available;
- Trojan client/server;
- standalone UDP;
- TLS-wrapped listeners/upstreams;
- reverse server/client after B1 or the minimal A1 repair;
- Linux transparent path in an environment-gated job.

## Workstream 6: comparison and reporting

Produce a machine-readable result containing:

- pass/fail/skip/error;
- pproxy and Eggress observations;
- normalized diff;
- raw artifact references;
- timing values and tolerances;
- capability/composition IDs;
- divergence IDs;
- environment metadata.

Generate a human report grouped by capability and composition. The parity manifest/report generator should be able to verify that every release-blocking drop-in entry has a passing scenario in an accepted environment.

Add failure minimization where practical: rerun with simplified payloads, fewer chain hops, or reduced fragmentation to produce a concise reproduction. Minimization must be bounded and optional.

## Workstream 7: CI integration

Create tiers:

1. **Fast structural:** schema validation, runner unit tests, fixture tests.
2. **Core differential:** Linux, pinned pproxy, common HTTP/SOCKS/CLI scenarios.
3. **Extended differential:** UDP, TLS, Shadowsocks, Trojan, reverse, routing.
4. **Platform differential:** macOS/Windows/Unix-specific subsets.
5. **Privileged/external:** transparent proxying, packet capture, independent implementations.

Preserve explicit environment gates for local development, but release workflows must run all required tiers. CI artifacts for failures should include redacted reports and logs.

## Testing requirements

- unit tests for schema, normalization, timeout, output bounds, and comparison logic;
- process-guard tests proving cleanup on drop, timeout, panic, and cancellation;
- fake oracle fixtures for deterministic runner testing;
- successful two-sided runs for representative TCP and UDP scenarios;
- Windows compile/tests for process handling;
- redaction tests for credentials in argv, URI, env, logs, and artifacts;
- flake resistance: repeated core suite runs without port/process leakage.

## Acceptance criteria

- Scenarios are declarative, versioned, validated, and mapped to A2 IDs.
- The runner never relies on pproxy console output for readiness.
- Child processes and sockets are reliably cleaned up on every termination path.
- Core HTTP/SOCKS/CLI scenarios run automatically in CI against pinned pproxy.
- Extended scenarios generate reusable machine-readable reports.
- Every release-blocking drop-in capability references a passing oracle scenario or an approved evidence exception.
- Failures retain enough redacted evidence for reproduction.
- Timing comparisons use explicit tolerances and do not create routine flakes.
- Manifest/report consistency checks fail when evidence is stale or absent.

## Out of scope

- implementing missing protocols;
- promoting a capability based solely on a new scenario;
- benchmarking throughput at scale beyond the timing needed for semantic comparison;
- replacing independent interoperability tests where pproxy cannot serve as a meaningful oracle.

## Handoff notes

Keep scenario definitions separate from runner code. Extend the existing `eggress-testkit` differential/oracle modules rather than creating a parallel harness. Favor typed observations over ad hoc string comparisons. Any normalization that can hide a real semantic difference must be narrowly scoped and documented.
